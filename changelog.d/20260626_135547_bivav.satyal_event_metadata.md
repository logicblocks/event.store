### Added

- Added an optional `metadata` field to `NewEvent` and `StoredEvent`, backed by a `metadata JSONB NOT NULL` column in the `events` table. The field accepts an arbitrary JSON-serialisable value and defaults to `None` (no metadata), which is stored as JSON `null`.

### Changed

- The `serialise()` output of both `NewEvent` and `StoredEvent` now includes a `metadata` key. Consumers that persist or strictly validate serialised events (e.g. snapshot stores) should account for the new key.

### Breaking changes

- **`StoredEvent.metadata` is now a required constructor argument.** Code that constructs `StoredEvent(...)` directly (custom adapters, fixtures, tests) must pass `metadata` explicitly (use `None` for no metadata).
- **`NewEvent.metadata` is now a required constructor argument.** Code that constructs `NewEvent(...)` directly (custom adapters, fixtures, tests) must pass `metadata` explicitly (use `None` for no metadata).

- **A schema migration is required for existing deployments.** The `events` table gains a `metadata JSONB NOT NULL` column, and both the read and write paths now depend on it (the read path maps the column directly onto `StoredEvent.metadata`). The migration must therefore be applied **before, or atomically with, deploying the new version** — running the new code against an un-migrated table fails on reads as well as writes.

  ```sql
  SET lock_timeout = '2s';
  ALTER TABLE events ADD COLUMN metadata JSONB NOT NULL DEFAULT 'null'::jsonb;
  ALTER TABLE events ALTER COLUMN metadata DROP DEFAULT;
  ```

  The transient `DEFAULT 'null'::jsonb` backfills all existing rows during the `ADD COLUMN` step, matching the application's representation of absent metadata (JSON `null`, deserialised as `None`) so historical and newly-written no-metadata rows read back identically. The default is then dropped so the column's final state matches the canonical no-default DDL, ensuring all new rows must supply an explicit value. In PostgreSQL 11+ adding a column with a constant default is a catalogue-only change (no table rewrite); the bounded `lock_timeout` guards against blocking behind a long-running transaction on a busy table.

  Rollback is non-destructive: the previous version ignores the extra column, so an application-only rollback can leave the column in place. Dropping the column (`ALTER TABLE events DROP COLUMN metadata`) permanently discards all stored metadata and should be avoided unless the data is no longer needed.
